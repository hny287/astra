Overview

Core security guarantee

Raw source code never leaves the customer environment. Only normalized UnifiedFinding[] JSON is transmitted over HTTPS with gzip compression. All AI inference runs inside the data plane container or in customer-controlled infrastructure (Ollama, AWS Bedrock). This is enforced at the architecture level, not just policy.

Goals & Non-Goals

Goals (shipped or in active development)

  • Single scan command running multiple security scanners in parallel, unified findings
  • Pluggable scanner architecture — add tools as pipeline nodes
  • AI enrichment runs inside data plane — raw code never sent externally without explicit opt-in
  • AI-inferred business logic rules with human confirmation and future-scan enforcement
  • Three role-specific dashboard workspaces (ADMIN / ANALYST / VIEWER)
  • Complete AI call observability — every call logged with tokens, latency, prompts, responses
  • Per-finding AI chat, per-scan AI chat, global AI assistant — all multi-turn with memory
  • Alert triage workflow with assignment, comments, history timeline
  • Export: JSON, CSV, SARIF, HTML, Markdown — all include triage fields
  • Env-driven branding — rebrand product by changing .env only, no code changes
  • Idempotent seed script, PrismaPg schema routing, trustHost for reverse proxy
  • Browser-side data store — stale-while-revalidate, zero redundant API calls

Non-Goals (explicitly out of scope for v1)

  • Runtime application security (RASP, IAST) — static analysis only
  • Mobile application scanning (iOS/Android binaries)
  • Fuzzing or dynamic analysis (DAST)
  • Multi-tenant SaaS — single-tenant per deployment in v1
  • CI platforms beyond GitHub, GitLab, Jenkins, generic Docker

Designed but not yet built (roadmap)

  • Streaming chat — spec written (Task 34), implementation pending
  • User-scoped config — 4-layer design (Task 10/35), not yet implemented
  • 9 new pipeline nodes — sbom, dependency_vuln, secret_scan, iac_scan, compliance, correlation, risk_score, report, notify
  • SSO / SAML / OIDC — architecture designed in original spec, not yet built
  • Jira / Slack / PagerDuty — integration contracts designed, not built
  • GitHub PR integration — inline comments, PR blocking, CI scan trigger

Scan Pipeline — Node Reference

The current pipeline has 6 nodes executing sequentially through an event-driven worker. Each node claims a job, executes, marks it complete, and enqueues the next node. 9 additional nodes are planned (Tasks 51–59).

NODE 01
Clone
git clone to /tmp/{APP_ID}-scan-{id}. Async fs/promises. Temp dir cleanup on any failure path via cleanupScanTmpDir().
✓ Complete
NODE 02
Discover
File enumeration, language detection, metadata extraction. Builds file map for downstream nodes. Uses DB prompts with buildDiscoverPrompt().
✓ Complete
NODE 03
Deep Scan
Per-file AI analysis + bundled scanner calls in parallel batches. singleFile param for per-alert rescan. Empty-response retry. Full enum constraints in prompt.
✓ Complete
NODE 04
Cross-File
Business logic analysis across file summaries. Current issue: context bomb — all summaries concatenated. Fix: cluster_summaries node planned (Task 22).
⚠ Context bomb
NODE 05
Aggregate
Merges findings from all nodes. Deduplication by fingerprint (SHA-256 from data plane). Severity and category normalization.
✓ Complete
NODE 06
Persist
Writes normalized findings to DB. Auto-creates FINDING_TRIAGE tasks for HIGH/CRITICAL/MEDIUM findings. Sets scan status: COMPLETED. Calculates durationSeconds.
✓ Complete

Improvements Needed

1 — Cross-file context window bomb (Task 22) — Critical

P1 architecture issue — breaks large repos

Current: The cross-file node concatenates ALL file summaries from the discover phase into a single AI prompt. On repos with 50+ files, this exceeds the model's context window (typically 8K–32K tokens), producing empty or garbage output silently.

Suggested fix — cluster_summaries approach:

  • Group files by directory; produce one summary per directory cluster
  • Build a hierarchical summary tree — cluster summaries → sub-cluster summaries → root summary
  • For cross-file analysis: inject only the top-N most relevant clusters (cosine similarity or BM25 retrieval against the analysis query)
  • Use a sliding window — analyze pairs of adjacent clusters to find cross-boundary flows
  • Token budget enforcement: before injection, count tokens using proper tokenizer and trim to model_context_window × 0.7

2 — User-scoped AI credentials (Tasks 11–15) — Critical

Current: All users share API keys from environment variables. There is no per-user credential isolation — every user on the platform uses the same OpenAI key, Ollama URL, etc.

Suggested 4-layer architecture (Task 10 design):

LayerTableDescriptionTask
System registryProviderDefinition + ProviderModelAdmin-managed: what providers/models are available to this org#11
User credentialsUserProviderCredentialPer-user encrypted API keys (Ollama URL+key, OpenAI key, AWS creds). AES-256-GCM encrypted at rest.#12
User defaultsUserConfigPer-user default chat provider/model and per-node scan config#13
Named presetsUserPreset"Quick Scan", "Deep Audit" etc. — saved named configurations per user#14

3 — Durable job queue (no Redis)

Current: In-memory event-driven worker (improved from polling in v2.5). Server restart drops all pending jobs — scans in progress are lost.

Suggested: Replace with Redis + BullMQ. Benefits: persistence across restarts, dead-letter queue, rate limiting per provider, retry with exponential backoff, per-job progress tracking, distributed workers (multiple server instances).

4 — Token estimation (Task 23)

Current: Math.ceil(text.length / 4) — crude character division. Under-counts dense code, over-counts UTF-8 multibyte.

Suggested by model family:

  • OpenAI models: tiktoken library — exact BPE token count
  • Anthropic Claude: Anthropic token counting API (beta endpoint)
  • Ollama / local models: Character-based heuristic calibrated per model — Llama family ≈ text.length / 3.7 for code, / 4.2 for prose

Security Model

P0 security audit — all issues resolved as of v2.5.0

All critical security issues identified in the initial audit are fixed. Every API route requires authentication. Rate limiting, AES-256-GCM encryption, and RBAC fully implemented. 5 additional unauthenticated routes found and fixed in v2.8.0.

Authentication & authorization

  • All API routes require auth — only /api/auth/* and /api/v1/health are public
  • Three roles: ADMIN (full), ANALYST (read/write scans+findings), VIEWER (read-only)
  • ADMIN-only: config GET/PUT, user management, DELETE operations
  • canWrite() enforced on all state-changing routes
  • requireScanOwnership() — non-admins see only their own scans
  • Non-admin findings filtered by scan.userId
  • Presets scoped — non-admin sees own + built-in presets only
  • Signup removed from public UI — ADMIN creates users via /settings/users

Data protection

  • GitHub tokens AES-256-GCM encrypted at rest before DB write
  • Backward-compatible decryption for legacy plaintext tokens
  • Rate limiting: 10/min login, 5/min signup (IP-based in-memory sliding window)
  • JWT sessions via NextAuth v5 — trustHost: true for reverse proxy / custom domain
  • Raw source code never leaves the data plane container
  • Scan temp dirs cleaned on completion and failure
  • Audit history on every finding mutation — userId, timestamp, action
  • Pino structured JSON logs to ./logs/{APP_ID}.log

Architecture Reviews

Scored across 9 dimensions on a 1–5 scale. Score methodology: 1 = critical issues, 3 = functional with gaps, 5 = production-excellent.

01 — Scan pipeline
4.0 / 5

Event-driven worker is solid. Async I/O throughout. Temp dir cleanup on all paths. Deductions: sequential node execution (no DAG), in-memory queue (jobs lost on restart), cross-file context bomb unresolved.

02 — AI integration
3.8 / 5

Multi-provider with per-node overrides, multi-turn chat, DB-backed prompts, empty-response retry, full observability. Deductions: shared API keys (no per-user isolation), 3 stub providers in UI, provider cache hash incomplete, token counting inaccurate.

03 — Security posture
3.5 / 5

All routes auth-gated, RBAC implemented, AES-256-GCM for sensitive data, rate limiting, scan ownership. Deductions: no SSO (enterprise blocker), no per-user credential isolation, no API key auth, no session revocation, 5 routes were unprotected until v2.8.

04 — Data architecture
3.2 / 5

Prisma 7 ORM with proper migrations, schema routing fixed. Full audit history on mutations. Deductions: no per-user credential tables, no SBOM/compliance models, Config table as a key-value blob limits queryability, no Redis for durable queue.

05 — Carbon UI fidelity
4.5 / 5

IBM Plex Sans throughout, 0px border-radius, sentence-case labels, correct letter-spacing 0.16px, IBM Blue accent used correctly (not overused), surface hierarchy (canvas → surface-1), dark/light theme persisted. Landing page v2 is fully compliant. Minor deductions: border shorthand warnings fixed in v2 (task 82), some components still have minor deviations.

06 — API design
4.0 / 5

Consistent REST conventions, pagination with parsePagination helper, RBAC on every route, scan ownership scoping. Deductions: no versioning strategy beyond /api/v1, no OpenAPI spec, no API key auth for programmatic access, some endpoints discovered unauthenticated (fixed in v2.5/v2.8).

07 — Observability
4.8 / 5

Every AI call logged — no truncation — provider, model, tokens in/out, latency, system prompt, user prompt, raw response. Per-call retry from UI. Pino structured JSON logging. Scan log per node. Near-perfect deductions: no Prometheus metrics, no distributed trace IDs correlating AI calls to scans.

08 — Scalability
2.5 / 5

Major gaps: In-memory job queue limits to single server instance. No Redis — cannot scale horizontally. Sequential pipeline — no parallelism. No caching layer. Provider API keys in env — cannot isolate per-tenant traffic. Viable for small teams; fundamentally blocked for enterprise scale.

09 — Code quality
4.0 / 5

TypeScript throughout, Prisma typed queries, shared utilities (parsePagination, rateLimit, encrypt, rbac), Pino logging replacing console.log, idempotent seed. Deductions: dead LangGraph code (Task 31), duplicated ScanChat/AiChatProvider logic, some 'system' userId hardcoding in history entries (Task 24).
