Foundation
Key Principles
Every design decision in astra-core flows from these six principles.
01
AI-native scanning
Every scanning decision flows through AI understanding. No regex engine, no separate pattern matcher runs alongside the AI.
02
Knowledge-driven
Rules, guidelines, and OWASP patterns are knowledge that feeds AI context — not separate scanning engines being executed.
03
Cumulative prompts
Start with a base security role prompt, stack built-in rules, stack custom rules on top. Every rule adds to AI understanding.
04
Multi-provider AI
Ollama (local + cloud), OpenAI, Anthropic Claude, Google Gemini. All implement the same provider interface. Swappable.
05
CLI-first
astra scan ./my-repo — single command, works locally, in CI, everywhere. Server mode is a PoC post-feature.
06
Configurable
Everything tunable via astra.config.json. Provider, model, token limits, severity filters, custom rules — all overridable per project.
Architecture
Scan Pipeline
Sequential pipeline architecture. Each scan flows through ordered stages. Easy to add, remove, or reorder stages.
01
Load Config
Parse astra.config.json + merge CLI flags. Zod validation. Resolve provider, model, token limits.
02
Load Knowledge
Read rules/, guidelines/, prompt templates, custom rules. Assemble AI knowledge context.
03
Source Resolution
Auto-detect local path vs git URL. Clone to /tmp if needed. Resolve to local directory on disk.
04
AI Discovery
AI reads directory tree, ranks files P0–P4 by security relevance. Allocates token budget per file.
05
Deep Scan
Per-file AI analysis. Batched by batchSize. Produces findings[] + FileSummary[] per file.
06
Cross-File
All FileSummary[] aggregated. AI reasons across files — auth paths, broken access control, data flows.
07
Aggregate + Output
Normalize severity, CWE/OWASP. SHA-256 dedup. Format: JSON, table, HTML, SARIF.

Scan Inputs
Source Resolution
Every scan resolves to a local directory on disk. The source.ts module auto-detects the input type.
Local Directory
Already on disk. No auth needed. Use case: local dev, CI/CD volume mounts.
$ astra scan ./my-project
$ astra scan /home/user/app
Git Repo URL
Clones to /tmp/astra-scan-xxxx/. HTTPS or SSH. Supports --branch flag.
$ astra scan https://github.com/org/repo
$ astra scan git@github.com:org/repo
$ astra scan https://gitlab.com/org/repo
--branch feature/auth-refactor
Cloud Repo API Post-PoC
Fetch via GitHub/GitLab API. Avoids full clone. Requires PAT / OAuth token.
$ astra scan github:org/repo
$ astra scan gitlab:org/repo

Example Rule File
rule NO_EVAL {
  severity: HIGH
  category: SAST
  cwe: CWE-95
  match: /eval\s*\(/ in javascript,typescript
  message: "eval() — risk of code injection"
}

rule NO_DANGEROUS_HTML {
  severity: MEDIUM
  category: SAST
  match: /dangerouslySetInnerHTML/ in typescript
  message: "React XSS risk"
}
How Rules Work

Rules in .astra files are not executed as regex.

They are parsed and included in the AI's system prompt as additional knowledge context. The match pattern tells the AI what to look for — the AI uses its own understanding to find it.

This means rules work across variations, refactors, and obfuscations that a regex would miss entirely.

Architecture
Sequential Pipeline
Simple, testable, easy to add/remove stages. Can evolve to plugin/event-driven later.
AI as only scanner
Yes — no regex engine
User requirement: AI-native tool. Patterns feed AI context, not executed separately.
AI-guided discovery
Replaces file walker
AI decides what matters based on repo structure, not hardcoded path patterns.
Language
TypeScript / Node.js
User choice. Strong AI SDK ecosystem across all four providers.
Config system
Zod + JSON
Type-safe validation, good developer experience, fail-fast on bad config.
Token limits
Per-model in config
Different models have very different limits. Must be configurable, not hardcoded.
Custom rules
.astra rule language
Human-readable, easy to extend, parsed into AI context — not executed as matchers.
Cumulative prompts
Stack all layers
Base + built-in rules + custom rules all stack. Every rule adds to AI understanding.
Model pulling
Cloud API key first
User explicitly requested cloud models via API key — no local model pulling required.
