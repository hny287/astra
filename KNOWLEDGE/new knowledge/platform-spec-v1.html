Overview

Core Design Principle

Raw source code never leaves the customer environment. Only structured UnifiedFinding[] JSON crosses the control/data-plane boundary over HTTPS. All AI inference runs inside the data plane or in customer-controlled infrastructure (Ollama local, AWS Bedrock).

Security Engineers
Deep audit workflows
Run scans, configure policies, triage alerts via AI chat, assign tasks, track remediation. Full observability into every AI call.
DevOps / Platform Teams
CI/CD integration
Embed scanning in GitHub Actions, GitLab CI, Jenkins, or generic Docker. Zero new CI infrastructure needed.
CISO / Leadership
Risk posture & reporting
Severity trends, compliance posture, remediation velocity. Executive PDF reports and SARIF export.

Goals & Non-Goals

Goals (v1–v2)

  • Single scan command running multiple security scanners in parallel, unified findings output
  • Pluggable scanner architecture — add proprietary or third-party tools
  • AI enrichment runs inside the data plane; raw code never sent to external AI without org opt-in
  • AI-inferred business logic rules, human confirmation, enforcement in future scans
  • Three role-specific dashboard workspaces (Executive, Engineering, Sec Ops)
  • Slack, Jira, PagerDuty, GitHub, GitLab, generic webhook integrations
  • Three deployment models: SaaS, self-hosted, hybrid
  • RBAC: ADMIN / ANALYST / VIEWER
  • Full observability of every AI call — tokens, latency, prompt, response, retry
  • AI chat at 3 levels: per-finding, per-scan, global security assistant
  • Export: JSON, CSV, SARIF, HTML, Markdown, PDF executive report
  • Branding fully configurable via environment variables — single .env change to rebrand

Non-Goals (v1 scope)

  • Runtime security (RASP/IAST) — static analysis only in v1
  • Mobile app scanning — iOS/Android not in scope
  • DAST / fuzzing — future roadmap
  • Multi-tenant SaaS — single-tenant per deployment in v1
  • CI beyond GitHub/GitLab/Jenkins/Docker — not supported

Slipped from v1 to roadmap

  • Streaming chat — spec written (Task 34), implementation pending
  • User-scoped provider credentials — design done (Tasks 10/35), not executed
  • DB-first context loading — partial (Task 80), in-memory shortcuts remain
  • Resumable/incremental scans — Task 49, not started
  • WebSocket streaming findings — Task 50, not started
  • SSO (SAML/OIDC) — enterprise requirement, not yet implemented

Scan Pipeline — Node Reference

Node 01
Clone
Git clone to /tmp/astra-scan-{id}. Async fs/promises. Token decrypted from GithubProfile. Cleanup on success and failure.
✓ Stable
Node 02
Discover
Walk repo file tree. Language detection by extension. Build file metadata map. Filter excluded dirs.
✓ Stable
Node 03
Deep Scan
Per-file: Semgrep SAST + AI analysis. Parallel batch execution. Token estimation for context window. Empty-response retry logic.
⚠ Token est. inaccurate
Node 04
Cross-File
Business logic inference across all files. Concatenates ALL file summaries into one prompt — context window bomb on large repos.
⚠ Context bomb
Node 05
Aggregate
Deduplicate findings by SHA-256 fingerprint (scanner+ruleId+file+line+code). Merge scanner findings with AI analysis.
✓ Stable
Node 06
Persist
Write UnifiedFinding[] to PostgreSQL. Auto-create FINDING_TRIAGE tasks for CRITICAL/HIGH/MEDIUM. Mark scan COMPLETED.
✓ Stable

AI observability — AiCallLog captures

  • Provider + model + SDK version — exact provenance of every inference call
  • Input tokens + output tokens + total tokens — cost attribution per scan node
  • Latency in milliseconds — wall-clock time from send to first byte
  • Temperature + thinking depth — all model parameters at call time
  • Raw system prompt — full system prompt sent (DB or fallback)
  • Raw user prompt — full user prompt including code context
  • Raw response text — untruncated; what the model actually returned
  • Pipeline node + source — which node triggered the call (deep_scan, cross_file, chat, rescan)
  • Retry count — if empty-response retry was triggered (text: "" with non-zero outputTokens)
  • Retry button in UI — /api/v1/ai-calls/[id]/retry re-executes with same params, shows new response inline

Suggested Improvements

Arch improvement 1 — User-scoped provider credentials

Design complete — execution not started

4-layer config architecture designed (Tasks 10, 35) but not implemented. All AI calls use shared environment variables. Users cannot configure their own API keys — major enterprise blocker.

Proposed 4-layer resolution order:

1. System Defaults
admin-set global config in Config table
→
2. UserProviderCredential
per-user API keys, encrypted AES-256-GCM
→
3. UserPreset
"Quick Scan", "Deep Audit" named configs
→
4. Scan.configJson
per-scan override (DB field exists, not exposed in UI)

New tables needed: ProviderDefinition, ProviderModel (system registry), UserProviderCredential (per-user keys), UserConfig (per-user defaults).

Arch improvement 2 — Pipeline parallelization

  • Discover can start immediately after Clone has the repo path (streaming handoff)
  • Deep scan files run in parallel batches with dynamic concurrency based on provider rate limits
  • File content caching between Discover and Deep Scan — avoid reading each file twice (Tasks 46–48)
  • Resumable scans — checkpoint after each node; resume from failure point without re-cloning (Task 49)
  • Streaming findings via WebSocket/SSE as deep_scan progresses, not just at Persist (Task 50)

Arch improvement 3 — Cross-file context windowing

The cross-file node concatenates all file summaries into one massive prompt. Three fix strategies:

  • Sliding window with overlap — process N files at a time, overlap for continuity
  • Graph-based similarity — cluster related files by import graph; cross-analyze within clusters
  • Chunked passes — one pass per subsystem, final consolidation pass

Arch improvement 4 — Accurate token counting

Replace Math.ceil(text.length / 4) with tiktoken (for OpenAI/Anthropic) or provider-specific tokenizer. Fixes context window allocation and prevents silent truncation in deep_scan for large files.

Security Model

P0 Security Complete — as of v2.5 / v2.8

All P0 vulnerabilities resolved. Every API route requires auth. Rate limiting, AES-256-GCM encryption, RBAC all operational. AuthJS middleware correctly allows /api/auth/* routes.

What's implemented ✓

  • All API routes require auth — only /api/auth/* and /api/v1/health are public
  • Roles: ADMIN (full), ANALYST (read/write scans+findings), VIEWER (read-only)
  • ADMIN-only: config GET/PUT, user management, DELETE operations
  • canWrite() on all state-changing routes; canAdmin() for privileged ops
  • requireScanOwnership() — non-admins see only their own scans and findings
  • Presets scoped: non-admin sees built-in + own only
  • GitHub tokens AES-256-GCM encrypted at rest
  • Rate limiting: 10/min login, 5/min signup per IP
  • JWT sessions via NextAuth v5; trustHost: true for reverse-proxy
  • Source code boundary: raw code never leaves customer CI environment
  • Temp scan directories cleaned on completion AND failure
  • Audit history on every Finding mutation: userId, timestamp, before/after

What's still missing ⚠

  • SSO (SAML/OIDC) — hard gate for enterprise deployments
  • mTLS — data plane → control plane is HTTPS but no mutual TLS
  • API key auth — no API key system for programmatic access; only session JWT
  • Audit log export — history in DB but no SIEM export or download
  • Per-user API key encryption — provider keys in env, not per-user encrypted
  • Session invalidation — no logout-all-sessions or token revocation
  • CSRF protection — not explicitly implemented beyond NextAuth built-in
  • Secret rotation — no automation for rotating ENCRYPTION_KEY or auth secrets

Fixed security issues (history)

  • v2.8: PATCH /rules, GET /rules, GET /scans/[id]/stream authenticated
  • v2.8: AuthJS middleware intercept fixed (was breaking /api/auth/* routes)
  • v2.5: Blanket /api/* middleware bypass removed
  • v2.5: GitHub tokens encrypted at rest
  • v2.5: Rate limiting on login and signup
  • v2.5: Non-admin users scoped to own scan data
  • v2.1: Session-based auth replaces hardcoded userId in all routes

UnifiedFinding schema (canonical TypeScript interface)

interface UnifiedFinding {
  // Identity
  id: string;                   // UUID v4
  fingerprint: string;          // SHA-256(scanner+ruleId+file+lineStart+codeSnippetHash) — first 32 hex chars
  scanner: "trivy" | "semgrep" | "gitleaks" | "bearer" | "checkov" | "bandit" | "ai_deep_scan" | "ai_cross_file";

  // Classification
  title: string;
  description: string;
  severity: "CRITICAL" | "HIGH" | "MEDIUM" | "LOW" | "INFO";
  category: "SAST" | "SCA" | "SECRETS" | "IaC" | "DATA_FLOW" | "AI" | "COMPLIANCE";
  cwe: string[];                // e.g. ["CWE-89", "CWE-564"]
  owasp: string[];              // e.g. ["A03:2021 - Injection"]

  // Location
  file: string;                 // relative path from repo root
  lineStart: number;
  lineEnd: number;
  codeSnippet: string;          // 3–10 lines around the vulnerability
  language: string;

  // AI enrichment (null until deep_scan node completes)
  aiExplanation: string | null;
  aiFix: string | null;
  exploitScore: number | null;  // 0.0 – 10.0
  exploitationScenario: string | null;

  // Triage state
  status: "OPEN" | "CONFIRMED" | "IN_PROGRESS" | "REMEDIATED" | "FALSE_POSITIVE" | "ACCEPTED_RISK";
  assignedToId: string | null;

  // Relations
  scanId: string;
  userId: string;               // scan owner
  taskId: string | null;        // linked task (bidirectional)

  // Raw output
  raw: Record<string, unknown>; // original scanner JSON, preserved for debugging

  // Timestamps
  createdAt: Date;
  updatedAt: Date;
}
