Overview

Core design principle
Raw source code never leaves the customer environment. Only structured UnifiedFinding[] JSON is transmitted to the control plane. All AI inference runs inside the data plane container or in customer-controlled infrastructure (Ollama, Bedrock). This is architecturally enforced — not a policy promise.

Astra is a closed-source, enterprise-grade application security platform that combines static analysis (SAST), software composition analysis (SCA), secret scanning, IaC scanning, sensitive data flow analysis, and AI-assisted business logic extraction into a single unified workflow. It is designed for three personas:

Persona 1
Security engineers
Run deep audits, configure policies, manage findings, triage alerts, and build custom rules. Full access to AI enrichment and per-finding chat.
Persona 2
DevOps / platform teams
Embed Astra into existing CI pipelines with a single command. No new infrastructure required. GitHub Actions, GitLab CI, Jenkins, or any Docker-capable CI.
Persona 3
CISO / security leadership
Track risk posture, compliance velocity, remediation trends, and team performance. Export executive summary reports in SARIF, HTML, Markdown, CSV.

Goals & Non-Goals

Goals (v1–v2)

  • Single scan command running multiple security scanners in parallel with unified findings
  • Pluggable scanner architecture — add proprietary or third-party tools
  • AI enrichment runs inside the data plane — raw code never sent externally unless org opts in
  • AI-inferred business logic rules from source code, presented for human confirmation
  • Three role-specific workspaces: Executive, Engineering, Sec Ops
  • Integrate with Slack, Jira, PagerDuty, GitHub, GitLab, and generic webhooks
  • Support three deployment models: SaaS, self-hosted, hybrid
  • Be fully closed-source — the compiled binary is the product
  • Gate features by license tier: Free, Pro, Enterprise
  • IBM Carbon Design System UI — consistent, accessible, enterprise-grade

Non-Goals (v1–v2)

  • Runtime application security (RASP, IAST) — static analysis only
  • Mobile application scanning (iOS/Android) — not in scope
  • Fuzzing or dynamic analysis (DAST) — future roadmap item
  • Multi-tenant SaaS — single-tenant per deployment in v1/v2
  • CI platforms beyond GitHub Actions, GitLab CI, Jenkins, Docker
  • Real-time collaboration (shared editing) on findings/tasks
  • SIEM integration — export only via CSV/JSON, no push to Splunk/Datadog yet

Scan Pipeline

Pipeline design
6-node event-driven pipeline. Each node is a separate async function. The worker claims a job, executes nodes sequentially (nodes 3 and 4 have internal parallelism), and marks the scan complete when all nodes succeed. Retry with exponential backoff on AI calls. Temp dirs cleaned on both success and failure.
NODE 01 · CLONE
Clone
git clone the repo. Authenticated via GitHub PAT (decrypted from DB). Shallow clone (depth=1) for speed. Creates temp dir: astra-scan-{scanId}.
✓ Complete
NODE 02 · DISCOVER
Discover
Walk the cloned repo. Detect languages, build PrioritizedFile[] sorted by security relevance (auth > payments > API > config > data). Builds SystemIntelReport for AI context.
✓ Complete
NODE 03 · DEEP SCAN
Deep Scan
Per-file AI analysis + bundled scanners in parallel batches (concurrency=5). AI enrichment per finding. FileSummary[] generated per file for cross-file node.
✓ Complete
NODE 04 · CROSS-FILE
Cross-File
Business logic, auth chains, PII flows across files. KNOWN ISSUE: concatenates ALL file summaries into one prompt — context window bomb on large repos.
⚠ Context bomb
NODE 05 · AGGREGATE
Aggregate
Deduplicate by SHA-256 fingerprint, merge scanner results, compute risk scores, rank findings.
✓ Complete
NODE 06 · PERSIST
Persist
Write findings to DB. Auto-create FINDING_TRIAGE tasks for CRITICAL/HIGH/MEDIUM findings. Mark scan COMPLETED. Clean temp dir.
✓ Complete

Improvements Needed

Pipeline improvements

1 — Fix the cross-file context window bomb (P0)

Current: cross_file node concatenates ALL FileSummary[] entries into a single prompt. A repo with 100 files at ~500 tokens per summary = 50,000 tokens in one prompt. This exceeds most model context windows and produces degraded or empty output.

Fix: Implement cluster_summaries node (Phase 7) before cross_file. Groups FileSummary[] by domain (auth, payments, API, config, data) using heuristic matching. Each cross_file call gets a focused, bounded context — typically 3–8 file summaries per cluster. This pattern is the fix established in the scan graph design.

2 — User-scoped config architecture (P1)

Current: Provider API keys read from env vars. All team members using the same deployment share the same API keys. No per-user key isolation. Provider factory: process.env[config.apiKeyEnv].

Fix: Implement 4-layer config stack: Credentials (per-user, encrypted in DB) → Org Defaults → User Presets → Per-Scan Override. UserProviderCredential table (Task #12), UserConfig table (Task #13). Provider factory reads from DB, not env vars.

3 — Pipeline parallelism and DAG execution (P2)

Current: All pipeline nodes run sequentially: clone finishes → discover starts → deep_scan starts. No intra-pipeline parallelism between independent stages.

Fix: DAG-based execution where nodes declare dependencies. Independent nodes run concurrently. Deep-scan files run in parallel batches with auto-scaled concurrency based on provider rate limits. Incremental checkpointing allows resuming from the failure node (Tasks #46–#49).

4 — Accurate token counting (P2)

Current: Math.ceil(text.length / 4) — simple character division. Off by 2× for code with many symbols, whitespace, Unicode.

Fix: Use tiktoken for OpenAI models, Anthropic's token counting API for Claude, character-based heuristics calibrated per model family for Ollama.

5 — Streaming findings to UI (P2)

Current: Findings only visible after the persist node completes. A 10-minute scan shows nothing until the last second.

Fix: WebSocket or SSE emit per batch in deep_scan. UI displays findings in real-time as each file completes. Spec written in Task 34 — implementation pending.

Security Model

Solid foundation
Every API route requires authentication. Rate limiting, encryption, and RBAC are fully implemented. The source code boundary (code never leaves customer environment) is architecturally enforced.

What's implemented

  • All API routes require auth — only /api/auth/* and /api/v1/health are public
  • Three roles: ADMIN (full), ANALYST (read/write scans+findings), VIEWER (read-only)
  • ADMIN-only: config GET/PUT, user management, DELETE operations
  • canWrite() enforced on all state-changing routes
  • requireScanOwnership() — non-admins see only their own scans
  • Non-admin findings filtered by scan.userId
  • GitHub tokens encrypted at rest with AES-256-GCM
  • Rate limiting: 10/min login, 5/min signup (IP-based, sliding window)
  • JWT sessions via NextAuth v5 with trustHost for reverse proxy
  • Audit history on every finding mutation (userId, timestamp, action)
  • Temp dirs cleaned on completion and failure (cleanupScanTmpDir)
  • Raw source code boundary: code never transmitted to control plane

What's missing

  • SSO (SAML/OIDC) — email/password only; enterprise deployments need Okta/Azure AD
  • User-scoped API key encryption — provider API keys in shared env vars
  • mTLS — data plane → control plane HTTPS but no mutual TLS
  • API key auth — no API key system for CI programmatic access
  • Audit log export — history in DB but no SIEM/export
  • Session invalidation — no logout-all-sessions or token revocation
  • Content Security Policy — CSP headers not verified
  • Dependency audit — no automated npm audit in CI
