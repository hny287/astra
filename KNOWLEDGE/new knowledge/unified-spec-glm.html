Volume I · System Architecture
Executive Summary
Astra is an enterprise-grade application security scanning platform combining SAST, SCA, secret scanning, IaC scanning, and AI-augmented business logic flaw detection. The system is split into a Control Plane (stateful, Node.js/Next.js) and a Data Plane (stateless Python/Node.js Docker container). Raw source code never leaves the customer environment — only normalized finding JSON crosses the boundary.
Trust Boundary — Non-Negotiable
What crosses: UnifiedFinding[] JSON (scanner, ruleId, file path, line numbers, severity, code snippet excerpt, CWE/OWASP tags, aiExplanation, aiFix, exploitScore, fingerprint). What NEVER crosses: raw source code, full file contents, dependency files, git history, private keys, env var values.

P10 — Rile — Rule Engine (20–40s, AI + data)

Infers reusable business logic security rules from all scan passes. CANDIDATE rules are stored for Sec Ops review. Once confirmed, they become permanent enforcement rules injected into all future scans.

Lifecycle:

Scan N:     Rule inferred → CANDIDATE (confidence: 0.85, evidenceFiles: [payments.ts, stripe.ts])
Human review: Security engineer confirms → CONFIRMED
Scan N+1:   Confirmed rule injected into scan prompt → rule violation → BusinessLogicFinding
Feedback:   CONFIRMED rules distributed via token validation response

The Data Moat: Each org's confirmed rules make Astra smarter specifically for that org's codebase. This is the Semgrep Assistant Memories equivalent — an org-specific learning layer that compounds over time. Incumbents cannot replicate this retroactively.

P11 — Triage (< 1s, logic)

Pure conditional routing. Counts CRITICAL findings and dispatches enrich_critical if criticalCount > 0; always dispatches finalize. Threshold configurable: triage.severityThreshold: CRITICAL | HIGH.

P11b — Enrich Critical (1–3 min, conditional AI)

The false positive eliminator. Re-reads specific files referenced in CRITICAL findings with full file content. A second AI pass with more context reduces the 78% false-positive rate that pure-LLM IDOR detection suffers from.

  1. Take all CRITICAL UnifiedFinding[]
  2. Re-read specific files (full content)
  3. AI evaluates: true positive? full exploit scenario? additional affected files? concrete remediation?
  4. True positives: enrich with CVSSv3, exploit steps, impact, remediation
  5. False positives: drop confidence → mark INFO

P12 — Finalize (5–15s)

Waits for enrich_critical (with maxWait timeout), collects all findings, deduplicates, normalises, and emits.

StepAction
1Wait for enrich_critical
2Collect all findings from all tracks
3SHA-256 dedup: SHA256(scanner + ruleId + file + lineStart)
4Severity normalisation: consistent enum across scanners
5Sort: CRITICAL → HIGH → MEDIUM → LOW → INFO
6Filter by --fail-on threshold
7HTTPS POST gzip-JSON → Control Plane /api/v1/scans
8Set exit code: 0 (clean), 1 (findings above threshold), 2 (infrastructure error)

Post-Scan: Auto-Fix & MCP

Agentic Auto-Fix Engine

Triggered by dashboard action or automatic for HIGH/CRITICAL findings. Model: Pixee-class — never auto-merges, human approval mandatory.

  1. Classify fix type: simple (single-file), complex (multi-file refactor), business-logic (needs human input)
  2. Fetch full file context
  3. AI generates fix using full context + UnifiedFinding + ai_fix seed
  4. Validate: AST parse, no syntax errors, no broken existing patterns
  5. Generate unified diff (minimal change, preserve code style)
  6. Create feature branch astra/fix-{fingerprint-short}
  7. Commit: security: fix {title} [{severity}]
  8. Open draft PR with AI explanation + remediation notes + dashboard link
  9. Human review mandatory before merge

P0
Authentication & Authorization  CRITICAL
Authentication and authorization are the gateway to everything. A single flaw here means an attacker doesn't need to exploit anything else — they just walk in. No other category has a higher blast radius. Review every auth check, every permission gate, every session mechanism.
AUTH-01
Missing Authentication
Endpoints that perform sensitive operations with zero auth check. The route exists, it works, and anyone can call it.
POST /admin/users — no auth middleware DELETE /api/data — accessible without token
AUTH-02
Broken Access Control
Authenticated user can access another user's data or perform actions above their role. IDOR, horizontal privilege escalation, forced browsing.
GET /api/invoices/:id — no ownership check PUT /users/:id — any user can edit any user
AUTH-03
Privilege Escalation
User can elevate their own privileges — self-promoting to admin, role stored in JWT payload not verified against DB, mass assignment vulnerabilities.
isAdmin stored in JWT, never verified in DB role: admin in POST body accepted as-is
AUTH-04
Insecure JWT / Token Handling
Weak signing secrets, algorithm confusion (HS256 vs RS256), missing expiry, tokens not invalidated on logout, sensitive data in payload.
secret: "secret" or short/guessable key alg: "none" accepted by verifier No token blacklist on logout
AUTH-05
Session Management
Session fixation, sessions not invalidated on logout, predictable session IDs, no idle timeout, sessions valid across password reset.
Session ID not regenerated after login Session persists after password change
AUTH-06
Weak Password Policies
No minimum length, no complexity, passwords stored in plain text or MD5/SHA1, no bcrypt/argon2, passwords logged in error messages.
crypto.createHash('md5').update(pass) No rate limit on /login endpoint
AUTH-07
OAuth / SSO Misconfigurations
Missing state parameter (CSRF), open redirect in callback, token leakage in URL, accepting tokens from wrong audience/issuer.
No state param in OAuth flow redirect_uri not validated server-side
AUTH-08
API Key / Token Management
API keys with no expiry, no scope, no rotation, logged in plaintext, returned in responses after creation, no revocation mechanism.
API key returned on every GET /me Token logged: console.log('Token:', token)
What AI specifically checks for
→ Every route definition — is auth middleware applied?
→ Ownership checks on resource reads/writes/deletes
→ Role/permission checks — server-side, not client-side
→ JWT secret strength and algorithm validation
→ Cross-file: auth defined in A, missing from B
→ Role stored in token vs verified against DB
→ Session lifecycle — creation, rotation, invalidation
→ Admin operations — are they truly gated?

P1
Injection Vulnerabilities  CRITICAL
Injection is when attacker-controlled input reaches an interpreter — SQL engine, OS shell, HTML renderer, LDAP server — and is executed as code or commands. Every place where user input flows into a query, command, or template is a potential injection point.
INJ-01
SQL / NoSQL Injection
String concatenation into SQL queries or NoSQL filter objects. Attacker can dump, modify, or delete the entire database, or bypass auth entirely.
SELECT * FROM users WHERE id = ${userId} db.find({ name: req.body.name }) — NoSQL
INJ-02
Command Injection
User input passed to exec(), spawn(), system() without sanitization. Attacker runs arbitrary OS commands as the application user.
exec(`convert ${filename} output.png`) subprocess.call(f"ping {host}", shell=True)
INJ-03
XSS (Cross-Site Scripting)
User input rendered in HTML without escaping. Attacker executes JS in victim's browser — steals cookies, tokens, performs actions as victim.
innerHTML = userInput dangerouslySetInnerHTML={{ __html: data }} res.send(`Hello ${req.query.name}`)
INJ-04
LDAP Injection
Unsanitized user input in LDAP filter strings. Attacker can bypass authentication or extract directory data.
filter: `(uid=${username})` — unescaped
INJ-05
Template Injection (SSTI)
User input rendered through a template engine (Jinja2, Handlebars, Twig). Can lead to RCE — attacker executes server-side code.
render_template_string(user_input) — Jinja2 Handlebars.compile(req.body.template)
INJ-06
GraphQL Injection / Abuse
No query depth/complexity limits (DoS), introspection enabled in production (exposes schema), batching attacks, field-level authorization missing.
No depth limit — recursive query bombs introspection: true in production config
INJ-07
Prototype Pollution
JavaScript-specific. Attacker pollutes Object.prototype via merge/clone functions. Can escalate to RCE or auth bypass depending on how polluted properties are used downstream.
merge(target, userObject) — recursive merge req.body.__proto__.isAdmin = true
What AI specifically checks for
→ String concatenation into queries, commands, HTML
→ Use of parameterized queries / prepared statements
→ eval(), exec(), spawn() with user-controlled args
→ Template rendering with user-provided content
→ GraphQL schema — depth limits, complexity, introspection
→ Object spread/merge operations on user input
→ Data flow: req.body/params → dangerous sink
→ ORM usage — raw query escape hatches
Scan nodes: deep_scan secrets_scan cross_file
P2
Secrets & Credential Exposure  CRITICAL
Hardcoded secrets are instantly exploitable. They're often committed once and live in git history forever — even if "removed." A single leaked Stripe key, AWS credential, or JWT secret compromises the entire connected system, not just this codebase.
SEC-01
Hardcoded API Keys
API keys, tokens, OAuth client secrets embedded directly in source code. Stripe, Twilio, SendGrid, GitHub PATs, etc.
const stripeKey = "sk_live_abc123..." TWILIO_TOKEN = "AC7f8e9d..."
SEC-02
Hardcoded Passwords
Database passwords, admin passwords, service account credentials in config files, source code, or scripts.
password: "admin123" in config.js DB_PASS = "prodpassword" in .env committed
SEC-03
Private Keys & Certificates
RSA/EC private keys, TLS certificates, SSH keys committed to the repository. Often in .pem, .key, or .p12 files.
-----BEGIN RSA PRIVATE KEY----- id_rsa committed to /config
SEC-04
Secrets in Git History
Secrets removed from current code but present in git history. Any clone of the repo exposes them. Requires full history scan, not just HEAD.
git log -p | grep "sk_live" Deleted .env still in history
SEC-05
Secrets in Logs / Error Responses
Secrets that don't live in code but get logged at runtime — tokens in URL params logged by request logger, auth headers in error dumps, debug mode exposing env vars.
logger.info(req.headers) — logs Authorization console.error(process.env) on startup
What AI specifically checks for
→ String literals matching key/secret patterns
→ Env var assignments with sensitive names
→ Config files with plaintext credentials
→ Git history scan (Gitleaks integration)
→ Logging statements that include auth headers
→ process.env exposure in error handlers
→ Base64-encoded secrets (common obfuscation)
→ Service-specific patterns: sk_live_, ghp_, AKIA
Scan nodes: secrets_scan deep_scan
P3
Business Logic Flaws  CRITICAL
Business logic flaws are the hardest category to find and the most valuable to attackers. They require understanding what the application is supposed to do — then finding cases where it doesn't enforce that correctly. No regex, no CVE database, no static pattern can find these. Only AI with system-level context can reason about them.
BIZ-01
Price & Quantity Manipulation
Client-controlled pricing, negative quantities, integer overflow in totals, discount stacking, coupon abuse, free trials that don't expire.
price sent from client, not fetched from DB quantity: -1 gives credit instead of charge
BIZ-02
Race Conditions
TOCTOU (Time-of-Check Time-of-Use) bugs. Concurrent requests that exploit a non-atomic check-then-act. Classic: double-spend, double-withdrawal.
Check balance → (race) → deduct balance Verify OTP → (race) → use OTP twice
BIZ-03
Workflow Bypass
Skipping required steps in a multi-step process. Going directly to step 5 of a 7-step checkout without completing steps 1-4.
POST /confirm-order without /validate-payment Email verified flag never checked before access
BIZ-04
State Machine Violations
Forcing an entity into an invalid state — cancelling an already-shipped order, re-using a one-time token, approving an already-approved request.
SHIPPED order can be CANCELLED via API Expired invitation token still accepted
BIZ-05
Limit & Quota Bypass
Rate limits only on one endpoint when the same action is available via multiple paths. API limits enforced client-side only. Free tier limits not enforced server-side.
Rate limit on /register but not /invite Storage limit checked on upload, not on copy
BIZ-06
Trust Boundary Violations
Trusting data from untrusted sources — accepting webhook payloads without signature verification, trusting X-Forwarded-For for IP-based auth, using client-provided user IDs.
Stripe webhook: no signature verification userId from req.body instead of req.user
What AI specifically checks for
→ Prices/amounts: sourced from client or DB?
→ Multi-step flows: are all steps enforced server-side?
→ State transitions: what's the valid state machine?
→ Concurrent operations: are they atomic/transactional?
→ Webhook handlers: is signature verification present?
→ Rate limits: consistent across all paths to same action?
→ User IDs: from authenticated session, not request body?
→ Cross-file: does the intended flow match what's enforced?
Scan nodes: cross_file enrich_critical deep_scan
P4
Cryptography & Data Protection  HIGH
Using cryptography incorrectly is as dangerous as not using it at all. MD5 and SHA1 are broken for security purposes. ECB mode leaks patterns. Short keys are brute-forceable. These are easy to scan for because the bad choices are well-known.
CRYPTO-01
Broken Hash Algorithms
MD5, SHA1 used for password hashing or integrity verification. Both are cryptographically broken — collision-vulnerable and rainbow-table vulnerable.
hashlib.md5(password.encode()) crypto.createHash('sha1').update(pass)
CRYPTO-02
Weak Encryption
DES, 3DES, RC4, ECB mode, short key lengths (<128-bit AES). ECB mode is especially dangerous — it leaks data patterns in the ciphertext.
createCipheriv('aes-128-ecb', key, '') DES.encrypt(data, key)
CRYPTO-03
Insecure Randomness
Math.random() for security tokens, session IDs, OTPs, or CSRF tokens. Math.random() is not cryptographically secure — it's predictable.
token = Math.random().toString(36) random.random() for OTP generation
CRYPTO-04
Missing Encryption at Rest/Transit
PII, financial data, health data stored in plaintext in DB or files. HTTP endpoints for sensitive operations. TLS disabled or version too old (TLS 1.0/1.1).
ssn TEXT NOT NULL — no encryption rejectUnauthorized: false in TLS config
CRYPTO-05
Improper Key Management
Encryption keys stored next to encrypted data, hardcoded IV/nonce, IV reuse, keys not rotated, master key accessible to application directly.
iv = Buffer.alloc(16, 0) — zero IV Key stored in same DB as encrypted data
What AI specifically checks for
→ Hash function names: MD5, SHA1, SHA-256 for passwords
→ Password hashing: bcrypt/argon2/scrypt only
→ Cipher mode: ECB = instant flag
→ Math.random() / random.random() for security uses
→ TLS configuration — version, cert validation
→ IV/nonce generation — must be random, not zero/static
→ Sensitive fields in DB schema — encrypted or not
→ Key storage location vs encrypted data location
Scan nodes: deep_scan cross_file
P5
Input Validation & Sanitization  HIGH
Every piece of data that enters the system from outside — request body, query params, headers, file uploads, webhooks — must be treated as hostile. The attack surface is everything you accept. Failure to validate leads to the full spectrum from minor information disclosure to full server compromise.
VAL-01
Path Traversal
User-controlled file paths used in file operations without normalization. Attacker reads /etc/passwd, source code, or writes arbitrary files.
readFile(`./uploads/${req.params.file}`) ../../../etc/passwd traversal
VAL-02
SSRF (Server-Side Request Forgery)
Application fetches a URL provided by the user. Attacker can probe internal services, AWS metadata endpoint, or exfiltrate data via DNS.
fetch(req.body.webhookUrl) — unvalidated curl http://169.254.169.254/latest/meta-data/
VAL-03
Open Redirect
Application redirects to user-controlled URL without validation. Used in phishing and to steal OAuth tokens by chaining with auth flows.
res.redirect(req.query.next) — unvalidated 302 Location: https://attacker.com
VAL-04
Mass Assignment
ORM/model accepts all fields from request body without allowlist. Attacker sets isAdmin, role, price, or other protected fields directly.
User.create(req.body) — no field filter Object.assign(user, req.body)
VAL-05
File Upload Vulnerabilities
No file type validation, MIME type trusted from client, files stored in web-accessible directory, no size limits, executable files accepted.
No mimetype check on file uploads File stored in /public/uploads/ → RCE
VAL-06
XXE (XML External Entity)
XML parser configured to process external entities. Attacker reads local files, performs SSRF, or causes DoS via billion laughs attack.
xml2js with no entity expansion limit DOCTYPE with SYSTEM entity in user XML
What AI specifically checks for
→ File path operations with user input — path.join safety
→ fetch/axios/http calls with user-provided URLs
→ Redirect destinations — validated allowlist or not
→ ORM .create() / .update() — explicit field selection
→ File upload handlers — type, size, storage location
→ XML parsers — entity expansion configuration
→ Schema validation — where does it happen, what's missing
→ Input validation: present at API boundary, not just UI
Scan nodes: deep_scan cross_file
P6
Dependency & Supply Chain  HIGH
Modern applications are mostly dependencies. Your own code might be perfect; a vulnerable library you import is still your problem. Supply chain attacks are increasing in frequency and sophistication — a malicious package can be indistinguishable from a legitimate one.
DEP-01
Known CVE in Dependencies
Packages with published CVEs at HIGH or CRITICAL severity. npm audit, Snyk, Trivy all catch these. Age of the CVE matters — old unpatched = intentional negligence.
lodash <4.17.21 — prototype pollution log4j 2.x — Log4Shell (CVE-2021-44228)
DEP-02
Unpinned / Floating Dependencies
^1.2.3 or * in package.json allows automatic updates to potentially malicious versions. Lockfiles not committed means different builds get different packages.
"axios": "*" — any version package-lock.json in .gitignore
DEP-03
Typosquatting / Malicious Packages
Packages with names similar to popular packages (lodahs, expres, cros). AI can flag suspicious package names that don't match known legitimate packages.
import 'cros' instead of 'cors' require('mongoos') instead of 'mongoose'
DEP-04
Outdated Runtime / Framework
Node.js 14 (EOL), Python 2.7, Django 2.x, Rails 5.x — EOL runtimes get no security patches. Framework version determines which vulnerability classes are possible.
engines: { node: "14.x" } — EOL Django==2.2.28 in requirements.txt
What AI specifically checks for
→ Package names vs. known legitimate packages
→ Version constraints — pinned vs. floating
→ Lockfile presence and integrity
→ Runtime version in engines / .nvmrc / Dockerfile
→ Direct CVE database lookup via Trivy integration
→ Transitive dependencies for critical packages
→ Deprecated packages still in use
→ Dependency count — massive dep trees = large attack surface
Scan nodes: iac_scan deep_scan
P7
Infrastructure & Configuration  MEDIUM
IaC misconfigurations are the modern equivalent of leaving the server room unlocked. Terraform files, Helm charts, Dockerfiles, and K8s manifests define the actual security posture of production — and they're code, which means they can be reviewed.
INFRA-01
Overprivileged IAM
S3 buckets with public ACLs, IAM roles with * actions or * resources, Lambda functions with AdministratorAccess. Violates principle of least privilege.
Action: "*", Resource: "*" in IAM policy aws_s3_bucket_acl { acl = "public-read" }
INFRA-02
Network Exposure
Security groups open to 0.0.0.0/0, databases accessible from internet, SSH/RDP open to world, no VPC, services binding to 0.0.0.0 in containers.
cidr_blocks = ["0.0.0.0/0"] on port 22 RDS publicly_accessible = true
INFRA-03
Container Security
Running as root in container, privileged: true, host network/PID sharing, no resource limits, mounting sensitive host paths, base images with known CVEs.
USER root in Dockerfile securityContext: { privileged: true }
INFRA-04
Insecure Defaults
Debug mode in production, default admin credentials, CORS wildcard (*), missing security headers (CSP, HSTS, X-Frame-Options), directory listing enabled.
DEBUG=True in Django production cors({ origin: '*' }) — all origins
INFRA-05
Missing Encryption at Infrastructure Level
S3 buckets without server-side encryption, RDS without encryption, EBS volumes unencrypted, secrets in environment variables instead of secrets manager.
aws_s3_bucket — no server_side_encryption ENV DB_PASS=hardcoded in Dockerfile
What AI specifically checks for
→ IAM policies — * actions/resources, admin roles
→ Security group rules — 0.0.0.0/0 on sensitive ports
→ Dockerfile — USER root, privileged, sensitive mounts
→ K8s manifests — securityContext, resource limits
→ CORS configuration — origins, methods, credentials
→ Security headers — CSP, HSTS, X-Frame-Options
→ Debug/development flags in production configs
→ Secrets in env vars vs secrets manager references
Scan nodes: iac_scan deep_scan
P8
Error Handling & Logging  MEDIUM
Bad error handling doesn't directly compromise systems but it provides attackers with the reconnaissance they need. Stack traces reveal frameworks, library versions, and file paths. Logs that contain PII or tokens create liability and enable further attacks. Silent failures hide real security events.
ERR-01
Information Leakage in Errors
Full stack traces returned to client, internal paths exposed, database error messages with schema info, framework/version disclosure in headers.
res.status(500).json({ error: err.stack }) X-Powered-By: Express 4.18.2
ERR-02
PII / Secret Logging
Passwords, tokens, credit card numbers, SSNs, email addresses logged. Authorization headers logged by request middleware. Health data in debug logs.
logger.debug('Request:', req.headers) — logs tokens console.log('User:', JSON.stringify(user)) — PII
ERR-03
Missing Audit Trail
No logging of auth events (login, logout, password reset), privilege changes, sensitive data access, admin actions. Compliance requirement and forensic necessity.
Admin deletes user — no log entry Failed login attempts not recorded
ERR-04
Silent Failures
Catch blocks that swallow exceptions, empty catch{}, security checks that fail open instead of closed. An auth check that throws is treated as "auth passed."
try { verifyToken() } catch(e) { } — silent if (err) return; // continues execution
What AI specifically checks for
→ Error responses — stack traces, internal paths returned
→ Response headers — framework/version disclosure
→ Logger calls — what fields are being logged
→ Auth/admin operations — are they being audited
→ Empty catch blocks or catch blocks that continue
→ Security checks that throw vs return false
→ PII fields (email, ssn, dob) in log statements
→ GDPR/HIPAA data — logged anywhere?
Scan nodes: deep_scan cross_file
P9
Code Quality & Attack Surface  LOW
Code quality issues don't directly create vulnerabilities but they increase the probability of vulnerabilities hiding inside them. Dead code, debug endpoints, and overly complex functions are where bugs — including security bugs — tend to live undetected the longest.
QUAL-01
Dead Code & Zombie Endpoints
Unused routes, commented-out code with credentials, old API versions still active, test endpoints in production code that are never called but still accessible.
app.get('/debug/users') — test endpoint // OLD: password=admin123 (in comment)
QUAL-02
Overly Complex Functions
Functions over 200 lines, cyclomatic complexity over 15, deeply nested conditions. Complexity correlates with bug density — security bugs hide in complex code.
500-line auth handler — impossible to audit Nested if/else 8 levels deep in payment logic
QUAL-03
Unsafe Coding Patterns
Disabling security features with flags (NODE_TLS_REJECT_UNAUTHORIZED=0), TODO/FIXME comments near security code, using deprecated security APIs.
process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0' // TODO: add auth check here
QUAL-04
Missing Rate Limiting & DoS Protection
No rate limiting on auth endpoints, expensive operations (regex, PDF generation, image processing) callable without limits, no request size limits.
POST /login — no rate limit PDF generation endpoint — no auth, no limit
What AI specifically checks for
→ Routes with no apparent callers or references
→ TODO/FIXME comments mentioning security
→ TLS_REJECT_UNAUTHORIZED, strictSSL: false
→ Function length and nesting depth in security-critical files
→ Auth/payment endpoints — rate limiting middleware?
→ Commented-out code containing credentials or tokens
→ Deprecated API usage (Node.js crypto, Python hashlib)
→ Request body size limits on upload endpoints

Pipeline Architecture Notes
Nonlinear DAG: Three intelligence nodes (A·B·C) fan out from Repo Intel and converge at Scan Orchestrator. Four scan tracks run in parallel. Deep-scan merges all tracks before cross-file analysis splits into per-domain clusters then converges globally. Triage conditionally routes CRITICAL findings to enrich-critical before finalization.

Data sovereignty: Source code never leaves the customer environment (Data Plane). Only normalized finding JSON crosses the trust boundary via HTTPS POST to the Control Plane.

Feedback loop: Confirmed BusinessLogicRules from the Control Plane are distributed back to the Data Plane via token validation, making each successive scan smarter for that organization.
