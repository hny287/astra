BRD — Business Requirements Document

1. Problem Statement

Enterprise security teams run 5+ separate scanning tools (SAST, SCA, secret scanning, IaC, DAST) with no unified view. Findings use different schemas, severity scales, and reporting formats. Triage is manual, slow, and error-prone. Business logic flaws—missing auth, privilege escalation, broken access control—cannot be detected by rule-based scanners at all.

2. Business Objectives

Unified Findings

Single normalized schema across all scanners. One dashboard instead of 5.

AI-Augmented Triage

AI explains every finding in plain English, suggests fixes, and scores exploitability.

Business Logic Detection

AI infers application rules, detects violations, and presents them for human confirmation.

Source Code Never Leaves

Data Plane runs in customer infrastructure. Only findings JSON crosses the boundary.

CI/CD Native

Single Docker command integrates with GitHub Actions, GitLab CI, Jenkins, and any CI platform.

Compliance Ready

Audit logging, RBAC, SLA enforcement, data residency controls for SOC2/ISO27001.

5. Business Rules

  • Raw source code must NEVER leave the customer environment
  • All findings must be deduplicated before storage (SHA-256 fingerprinting)
  • CRITICAL findings must trigger alerts within 5 minutes of ingestion
  • SLA timers must start immediately upon finding ingestion
  • AI-generated rules must be CANDIDATE status until human confirmed
  • Audit events must be retained for minimum 7 years
  • Org data isolation must be enforced at database level (row-level security)
  • Feature gates: Free (3 scanners, no AI), Pro (all scanners + AI), Enterprise (business logic + compliance)

PRD — Product Requirements Document

1. Product Overview

Astra is a closed-source security scanning platform with two operating modes: CLI mode (docker run --rm for CI pipelines) and API mode (long-lived HTTP server for self-hosted deployments). It runs 6 scanner binaries in parallel, enriches findings with AI, and emits normalized results to a Control Plane.

2.2 AI Enrichment

Layer 1: Per-File Deep Scan

Each code file is sent to AI for vulnerability analysis. Prompt: "Analyze this file for vulnerabilities, logic bugs, anti-patterns." Returns findings[] + file_summary per file. Parallel via batching.

Layer 2: Cross-File Business Logic

Aggregated file summaries form a codebase map. AI identifies cross-file issues: missing auth middleware, privilege escalation, broken access control, data flow violations. Returns biz_logic_findings[]. Serial after Layer 1.

2.6 CI/CD Integration

Universal Docker-based integration. The public action wrapper contains zero business logic — it simply pulls and runs the closed-source scanner image.

Universal Docker Command
# Works with GitHub Actions, GitLab CI, Jenkins, Azure DevOps, CircleCI, any Docker-capable CI docker run --rm \ -v $(pwd):/repo \ -e MODE=cli \ -e ASTRA_TOKEN=${ASTRA_TOKEN} \ -e OLLAMA_API_KEY=${OLLAMA_API_KEY} \ ghcr.io/astra/scanner:latest \ astra-scan --repo /repo --branch ${BRANCH_NAME}

2. Database Schema

-- Scans table: tracks each scan execution CREATE TABLE scans ( id INTEGER PRIMARY KEY AUTOINCREMENT, repo_url TEXT NOT NULL, branch TEXT NOT NULL DEFAULT 'main', commit_sha TEXT, triggered_by TEXT NOT NULL DEFAULT 'manual', scanners_run TEXT NOT NULL DEFAULT '[]', -- JSON array duration_seconds INTEGER, status TEXT NOT NULL DEFAULT 'RUNNING', -- RUNNING|COMPLETED|FAILED created_at DATETIME DEFAULT CURRENT_TIMESTAMP ); -- Findings table: normalized results from all scanners + AI CREATE TABLE findings ( id INTEGER PRIMARY KEY AUTOINCREMENT, fingerprint TEXT NOT NULL, -- SHA-256 dedup key scan_id INTEGER NOT NULL REFERENCES scans(id), scanner TEXT NOT NULL, rule_id TEXT NOT NULL, title TEXT NOT NULL, description TEXT NOT NULL DEFAULT '', severity TEXT NOT NULL, -- CRITICAL|HIGH|MEDIUM|LOW|INFO category TEXT NOT NULL, -- SAST|SCA|SECRETS|IAC|DATA_FLOW|BUSINESS_LOGIC file TEXT NOT NULL, line_start INTEGER NOT NULL DEFAULT 0, line_end INTEGER NOT NULL DEFAULT 0, code_snippet TEXT NOT NULL DEFAULT '', language TEXT NOT NULL DEFAULT '', cwe TEXT NOT NULL DEFAULT '[]', -- JSON array owasp TEXT NOT NULL DEFAULT '[]', -- JSON array ai_explanation TEXT, -- AI-generated ai_fix TEXT, -- AI-generated fix code exploit_score REAL, -- 0-10 remediation TEXT NOT NULL DEFAULT '', raw TEXT NOT NULL DEFAULT '{}', -- Original scanner JSON created_at DATETIME DEFAULT CURRENT_TIMESTAMP, UNIQUE(fingerprint, scan_id) ); -- Business logic rules: AI-inferred, human-confirmed CREATE TABLE biz_logic_rules ( id INTEGER PRIMARY KEY AUTOINCREMENT, scan_id INTEGER NOT NULL REFERENCES scans(id), rule_text TEXT NOT NULL, confidence REAL NOT NULL, -- 0.0-1.0 evidence_files TEXT NOT NULL DEFAULT '[]', status TEXT NOT NULL DEFAULT 'CANDIDATE', -- CANDIDATE|CONFIRMED|REJECTED violation_description TEXT, created_at DATETIME DEFAULT CURRENT_TIMESTAMP );

4. Two-Layer AI Architecture

Layer 1: Per-File Deep Scan

Input: Individual source files
Steps: File Walker → Chunker → Batch by language → AI Model
Prompt: "Analyze this file for vulnerabilities, logic bugs, anti-patterns"
Output: findings[] + file_summary per file

Parallel · Handles 5000+ line files via chunking (function/line/AST-based)

Layer 2: Cross-File Business Logic

Input: Aggregated codebase map (all file summaries)
Steps: Build structured graph → AI Model
Prompt: "Identify cross-file: missing auth, privilege escalation, broken access control, data flow violations"
Output: biz_logic_findings[]

Serial · After Layer 1 · Confidence-scored · CANDIDATE until human confirmed

Glossary

Astra
The security scanning platform. Named for the Greek word for "star" — illuminating hidden vulnerabilities.
Control Plane
Stateful backend (Node.js/Fastify) that ingests findings, enforces policies, manages integrations, and serves the dashboard. Hosted by Astra (SaaS) or customer (self-hosted).
Data Plane
Stateless Python Docker container that executes scanner binaries, runs AI analysis, normalizes findings, and emits results. Runs in customer CI/CD infrastructure. Raw source code never leaves this plane.
Unified Finding Schema
Normalized data model that maps findings from all scanners (Trivy, Semgrep, Gitleaks, Bearer, Checkov, Bandit, AI) to a single consistent structure with severity, category, CWE, OWASP, AI explanation, fix, and exploit score.
Fingerprint
SHA-256 hash of (scanner + ruleId + file + lineStart + codeSnippet). Used for deduplication — identical findings from different scans are merged via UPSERT.
Severity
CRITICAL (exploitable now), HIGH (significant risk), MEDIUM (moderate risk), LOW (minor issue), INFO (informational). Mapped from each scanner's native severity scale.
Category
SAST (static code analysis), SCA (dependency vulnerabilities), SECRETS (exposed credentials/keys), IAC (infrastructure misconfigurations), DATA_FLOW (sensitive data paths), BUSINESS_LOGIC (cross-file security rule violations).
Business Logic Rule
An AI-inferred security rule about how the application should behave (e.g., "All admin routes must have auth middleware"). Stored as CANDIDATE until human confirms or rejects. Confidence score 0.0-1.0.
AI Layer 1
Per-file deep scan. Each source file is sent individually to the AI model for vulnerability analysis. Returns findings[] + file_summary per file.
AI Layer 2
Cross-file business logic analysis. Aggregated file summaries form a codebase map. AI identifies cross-file security issues that single-file analysis cannot detect.
Enrichment
AI-generated addition to a finding: plain-English explanation, concrete fix code snippet, and exploitability score (0-10). Applied to HIGH/CRITICAL findings.
Exploit Score
AI-assessed 0-10 score indicating how easily a vulnerability could be exploited. Higher = more urgent. Used for SLA prioritization and auto-routing.
SLA
Service Level Agreement timer. Starts when a finding is ingested. CRITICAL: 4h, HIGH: 24h, MEDIUM: 72h, LOW: 7 days. Exceeded SLAs trigger escalation.
RBAC
Role-Based Access Control. Four roles: Admin (full access), SecOps (triage+policies), Engineer (view+fix), Viewer (read-only). Enforced per API request.
Ollama
AI inference server (local or cloud). Supports models like llama3, deepseek-coder, kimi-k2.6:cloud, GLM5.1:cloud. Astra uses OLLAMA_HOST, OLLAMA_MODEL, and OLLAMA_API_KEY environment variables.
CLI Mode
docker run --rm execution for CI/CD pipelines. Container starts, scans, emits findings, exits. Stateless. No persistent storage needed in the container.
API Mode
Long-lived HTTP server mode for self-hosted deployments. FastAPI/Uvicorn serves scan requests over HTTP. Planned for production.
CWE
Common Weakness Enumeration. Standardized vulnerability identifier (e.g., CWE-89 = SQL Injection, CWE-798 = Hardcoded Credentials).
OWASP
Open Web Application Security Project. OWASP Top 10 categorizes the most critical web app security risks (e.g., A03:2021 = Injection).
Trivy
Go-based scanner by Aqua Security. Scans for CVEs (vulnerabilities), IaC misconfigurations, and exposed secrets. Category: SCA/IaC/SECRETS.
Semgrep
Python-based SAST scanner. Pattern-based code analysis with 500+ community rules. Category: SAST.
Gitleaks
Go-based secret scanner. Detects API keys, passwords, tokens in Git history and working tree. Category: SECRETS.
Bearer
Go-based data flow scanner. Identifies sensitive data paths through application code. Category: DATA_FLOW.
Checkov
Python-based IaC scanner by Bridgecrew. Scans Terraform, Helm, Kubernetes, CloudFormation. Category: IaC.
Bandit
Python-based SAST scanner by OpenStack. Python-specific security issues. Category: SAST.
IBM Carbon Design System
IBM's open-source design system. Used for Astra's dashboard UI. Key tokens: IBM Plex Sans, 0px border radius, #0f62fe primary, weight 300 display type, 1px hairline borders, 0.16px letter-spacing.
Zero Trust
Security model where no request is trusted by default. Every request passes through WAF, TLS, auth, and API gateway before reaching any service. Data Plane is isolated from Control Plane.
